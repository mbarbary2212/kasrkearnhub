# KALMHUB — Interactive Clinical Cases

## Complete Documentation for Medical Educators & Administrators

---

# Part 1: How the System Works

## 1.1 Overview

Interactive Clinical Cases are structured OSCE-style simulations where a student works through a clinical scenario section by section. Each case is **pre-built** — generated by AI from chapter material and then reviewed/edited by faculty before publication. Students cannot skip ahead; they progress linearly and receive a detailed performance report at the end.

---

## 1.2 Case Structure

Every case can include up to **10 clinical sections**, plus an overarching **Professional Attitude** score. Admins choose which sections to activate when creating a case.

| # | Section | What the Student Does |
|---|---------|----------------------|
| 1 | **History Taking** | Interviews a virtual patient via text chat or voice (Egyptian Arabic). Asks open-ended questions; the AI patient responds in character. |
| 2 | **Physical Examination** | Taps on body regions (head/neck, chest, abdomen, limbs, etc.) to reveal clinical findings. Vitals are displayed in a structured grid. |
| 3 | **Investigations — Labs** | Orders lab tests from an available menu. Key tests earn points; unnecessary tests incur a **−1 penalty** each. |
| 4 | **Investigations — Imaging** | Orders imaging studies. Same key/unnecessary penalty logic as labs. |
| 5 | **Diagnosis** | Provides a possible diagnosis, differential diagnosis, and final diagnosis. Scored against expected answers with rubric-based evaluation. |
| 6 | **Medical Management** | Answers MCQs and/or free-text questions about medical treatment. |
| 7 | **Surgical Management** | Answers MCQs and/or free-text questions about surgical treatment. |
| 8 | **Monitoring & Follow-up** | Free-text response about ongoing monitoring plans. |
| 9 | **Patient & Family Advice** | Free-text response about counselling and discharge advice. |
| 10 | **Conclusion** | Tasks such as ward-round presentation, key clinical decisions, or learning points. |

**Professional Attitude** is scored holistically from the History Taking chat transcript — evaluating greeting, empathy, informed consent, and professional closure.

---

## 1.3 How AI Scoring Works

When the student completes all sections and submits, the entire case is sent to the `score-case-answers` edge function. Here is how each section type is evaluated:

### History Taking
- **Comprehension questions** (if the history mode is paramedic handover or triage note): Exact-match scoring against `correct_answer`.
- **Chat transcript**: AI evaluates which checklist items from categories A–E were adequately explored during the conversation.

### Physical Examination
- Scored on whether the student examined the relevant body regions. Points are awarded for regions examined that contain key findings.

### Labs & Imaging (Investigations)
- **Key tests/studies** (flagged `is_key: true`): Full points for ordering them.
- **Unnecessary tests**: Each non-key test ordered costs **−1 point** (penalty cannot go below 0).
- This teaches clinical efficiency — ordering everything is penalised.

### Diagnosis
- Three sub-rubrics: possible diagnosis, differential diagnosis, final diagnosis.
- Each is scored against `expected` answers and `expected_top` (primary answer).
- Partial credit for reasonable differentials; full marks for the correct final diagnosis.

### Management (Medical & Surgical)
- **MCQ questions**: Exact match against `correct` option.
- **Free-text questions**: AI compares the student's answer against `rubric.expected_points` and `rubric.model_answer`, awarding partial credit.

### Monitoring, Advice, Conclusion
- All use rubric-based free-text scoring: the AI checks how many `expected_points` the student addressed and compares against the `model_answer`.

### Professional Attitude
- Scored from the History Taking transcript (not a separate section the student sees).
- Items include: greeting the patient, introducing self, obtaining consent, showing empathy, summarising, and professional closure.
- Max score is typically 10 points.

---

## 1.4 Hints and Feedback

- **During the case**: Hints are embedded as "Consider including:" badges in free-text sections, showing 3 of the expected points (truncated) without revealing full answers.
- **After submission**: Each section displays AI-generated feedback explaining what was done well, what was missed, and the model answer.
- **Summary report**: Five exam-aligned categories are displayed:
  1. Professional Attitude
  2. History Taking
  3. Physical Examination
  4. Investigations (Labs + Imaging combined)
  5. Diagnosis & Management (Diagnosis + Medical Mgmt + Surgical Mgmt + Monitoring + Advice + Conclusion combined)
- A detailed 10-section breakdown is also available in an expandable view.

---

## 1.5 Voice Mode (History Taking)

Voice mode enables students to **speak** to the virtual patient in **Egyptian Arabic** and hear responses read aloud.

### How it works:
1. Student taps the microphone button.
2. **Web Speech API** (`webkitSpeechRecognition`) starts listening in `ar-EG` locale.
3. **Interim transcript** appears in real-time below the mic button as the student speaks.
4. Once the student finishes speaking, the final transcript is sent as a chat message.
5. The AI patient's response is read aloud using **SpeechSynthesis** at rate `1.1` (natural Egyptian Arabic pace).
6. `window.speechSynthesis.cancel()` is called before every new utterance to prevent audio overlap.

### Error handling:
- If the microphone is blocked → toast notification: "Microphone permission denied."
- If no speech detected → toast: "No speech detected — try again."
- After **2 consecutive failures**, a **fallback text input** automatically appears so the student can type in Arabic and still receive spoken AI responses.

---

## 1.6 Case Progression Rules

- **Linear flow**: Sections are presented in fixed order. The student cannot skip ahead.
- **Locked after submission**: Once a section answer is submitted, it cannot be changed.
- **Exit confirmation**: If the student tries to leave mid-case, a confirmation dialog appears warning that progress may be lost.
- **Anti-cheating measures**:
  - All case content has CSS `select-none` (text cannot be selected).
  - Copy, paste, and cut keyboard events are blocked.
  - Right-click context menu is disabled.
  - A semi-transparent **KALMHUB** watermark is overlaid on all case content to deter screenshot sharing.
  - Internal rubrics, model answers, and admin checklists are **never** sent to the student's browser — only the AI scoring function has access.

# Part 2: Admin Guide — Creating New Cases

## 2.1 Two Creation Pathways

| Pathway | Best For | What Happens |
|---------|----------|-------------|
| **Generate with AI** | Quickly creating cases from existing chapter material | The system reads the chapter PDF and any uploaded reference documents, then generates all 10 sections automatically. You review and edit before publishing. |
| **Build Manually** | Custom scenarios not tied to chapter content | An empty case skeleton is created with placeholder fields. You fill in every section by hand using the Case Preview Editor. |

---

## 2.2 Step-by-Step: Creating a Case

### Step 1 — Navigate
Go to the **Interactive** section in the admin panel and click **"Create Case"**.

### Step 2 — Tab 1: Basics
| Field | Description | Example |
|-------|-------------|---------|
| **Title** | Short, descriptive case title | "Acute Appendicitis in a 25-Year-Old Male" |
| **Chief Complaint** | The patient's presenting complaint | "Severe right lower abdominal pain for 6 hours" |
| **Module** | Which academic module this case belongs to | "General Surgery" |
| **Chapter** (optional) | Specific chapter within the module | "Chapter 4: Acute Abdomen" |
| **Level** | Difficulty: Beginner, Intermediate, or Advanced | "Intermediate" |
| **Estimated Minutes** | Expected completion time | 30 |
| **Additional Instructions** (optional) | Extra context for AI generation | "Focus on McBurney's point tenderness" |
| **Tags** | Searchable keywords | ["surgery", "acute abdomen", "appendicitis"] |

### Step 3 — Tab 2: Sections
- Toggle which of the 10 sections to include.
- For sections with questions (Medical/Surgical Management), set the number of questions (e.g., 3 MCQs + 1 free-text).

### Step 4 — Tab 3: History Mode
Choose how the patient history is delivered:

| Mode | Description |
|------|-------------|
| **Full Conversation** | Student interviews the patient from scratch via chat |
| **Paramedic Handover** | Student receives an ATMIST handover, then answers comprehension questions |
| **Triage Note** | Student reads a triage note, then answers comprehension questions |
| **Witness Account** | A witness describes what happened; student asks follow-up questions |
| **No History** | History section is skipped entirely |

Also choose the **Patient Language**: English or Egyptian Arabic.

### Step 5 — Tab 4: Patient
| Field | Description |
|-------|-------------|
| **Patient Name** | The virtual patient's name |
| **Age** | Patient age |
| **Gender** | Male / Female |
| **Avatar** | Choose from the avatar gallery |
| **Delivery Mode** | Practice (with hints) or Exam (no hints) |
| **Interaction Mode** | Text only or Text + Voice |

### Step 6 — Tab 5: Review & Create
Review all settings and click **Create**. Choose either:
- **"Generate with AI"** → AI creates all content (takes 30–60 seconds)
- **"Build Manually"** → Empty skeleton is created for manual editing

## 2.3 The Case Preview Editor

After creation, you land on the **Case Preview Editor** where you can review and modify every detail.

### What you can edit:
- **Case metadata**: Title, chief complaint, level, estimated time
- **Avatar**: Click the avatar to open the gallery picker
- **History interaction mode**: Toggle between text-only and voice
- **Active sections**: Toggle sections on/off (scores auto-recalculate)
- **Section content** (for each active section):
  - History checklist items (categories A–E)
  - Physical exam findings per body region
  - Lab tests (name, result, interpretation, key/not-key flag, points)
  - Imaging studies (same structure as labs)
  - Diagnosis rubric (expected answers, model answers, points)
  - Management MCQs (question, options, correct answer, explanation)
  - Management free-text (question, expected points, model answer)
  - Monitoring/Advice questions and rubrics
  - Conclusion tasks and rubrics
  - Professional attitude checklist items

### Publishing:
- Click **"Publish"** to make the case visible to students.
- Click **"Unpublish"** to hide it without deleting.

## 2.4 Defining Expected Answers

### MCQ Questions
- Provide 4 options (A, B, C, D).
- Mark exactly one as `correct`.
- Add an `explanation` that students see after submission.

### Free-Text Questions
Each free-text question has a `rubric` containing:
- **`expected_points`**: A list of key concepts the student should mention. Example:
  ```
  ["NPO status", "IV fluid resuscitation", "Broad-spectrum antibiotics", "Surgical consultation"]
  ```
- **`model_answer`**: A complete ideal answer. Example:
  ```
  "The patient should be kept NPO, started on IV normal saline for fluid resuscitation, given broad-spectrum antibiotics (e.g., ceftriaxone + metronidazole), and an urgent surgical consultation should be requested for possible appendectomy."
  ```
- **`points`**: Maximum points for this question.

The AI compares the student's response against both the expected points and the model answer, awarding partial credit proportionally.

### Diagnosis Rubric
Three sub-sections:
- **Possible diagnosis**: What conditions could this be? (`expected`: list of acceptable answers)
- **Differential diagnosis**: Ranked list of differentials (`expected`: list; `expected_top`: most likely)
- **Final diagnosis**: The definitive diagnosis (`expected`: single answer; `model_answer`: explained reasoning)

### Labs & Imaging
- Mark tests as **`is_key: true`** for tests the student should order.
- Leave `is_key: false` for tests that exist in the menu but are unnecessary for this case.
- Students lose 1 point for each unnecessary test ordered.

## 2.6 Lab Unit Convention

**All laboratory values must use Conventional (US) units.** Examples:

| Test | Correct Unit | ❌ Not This |
|------|-------------|------------|
| Glucose | mg/dL | mmol/L |
| Creatinine | mg/dL | µmol/L |
| Sodium | mEq/L | mmol/L |
| Hemoglobin | g/dL | g/L |
| WBC | ×10³/µL | ×10⁹/L |

This convention is enforced globally across all AI-generated cases. When editing manually, please follow the same convention.

# Part 3: External AI Generation Prompt

## How to Use This Prompt

1. Copy the prompt below.
2. Replace all `[BRACKETED PLACEHOLDERS]` with your case details.
3. Paste into Claude, ChatGPT, or any AI assistant.
4. Copy the JSON output from the AI response.
5. In KALMHUB, go to the Interactive Cases admin list and click **"Import JSON"**.
6. In the Import Modal, choose **"Paste Text"** and paste the JSON directly — no need to save a file.
   - Alternatively, save the output as a `.json` file and use **"Upload File"**.
7. Click **"Validate JSON"** then **"Import as Draft"**.

> **Tip:** If the AI output is long, you can copy the text directly from the chat and paste it into the KALM Hub Import Modal. The app automatically handles missing outer curly braces `{}`, markdown code fences, and trailing commas.
