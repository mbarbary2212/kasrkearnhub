Security Findings & Resolutions

1. Malicious Code Could Be Injected Into the Print Feature

Critical ✓ Fixed

What Was the Problem

The platform has a "Print" button on study documents that opens a printable version of the content. The way this worked, the document's title and content were copied directly into a new browser window without any safety checks. This meant that if a document contained hidden malicious code, that code would run automatically when anyone clicked Print.

Real-World Impact — What Could Have Happened

An attacker with admin access (or who had somehow inserted content into the platform) could craft a document with a specially designed title or body. The moment any student or staff member clicked the Print button on that document, the hidden code would silently run in their browser — stealing their login session, accessing their personal data, or performing actions on their behalf without their knowledge. This is known as a Cross-Site Scripting (XSS) attack.

What Was Fixed

The print feature was rewritten to build the printable page safely using browser-native methods that never interpret text as code. The document title is now always treated as plain text, and the content is cloned from the already-rendered page with any executable elements removed before printing.

2. Access Requests Could Be Forged as Already Approved

High ✓ Fixed

What Was the Problem

The platform has a public form where new users can submit an access request to join KasrLearnHub. The database rule that controls this form had no restrictions on what values could be submitted — it simply allowed anyone to insert any data. This included fields that should only be set by administrators, such as the request status and who reviewed it.

Real-World Impact — What Could Have Happened

Anyone with basic technical knowledge (no login required) could have submitted an access request directly to the database marking it as already "approved" and stamped with a fake admin reviewer. This would bypass the entire admin review process, potentially granting platform access to unauthorised individuals without any administrator ever seeing or approving the request. It could also be used to flood the admin inbox with thousands of fake requests, making it impossible to find genuine ones.

What Was Fixed

The database rule was updated to enforce that all new access requests must arrive in a "pending" state, with no reviewer assigned and no review date set. It is now impossible to submit a request that is already marked as approved or that pretends an admin has already reviewed it. Only an authenticated administrator can change the status of a request.

3. Any User Could See Private Video Watch Statistics

High ✓ Fixed

What Was the Problem

The platform tracks which students have watched each video lecture. There is a database view (a saved query) called "video view counts" that shows how many unique students have watched each video. This view was configured to bypass the platform's normal access controls — meaning it ran with administrator-level permissions regardless of who was asking for the data.

Real-World Impact — What Could Have Happened

Any logged-in student could query this view and retrieve aggregated viewing statistics across the entire student population — data that should only be visible to administrators and educators. While this particular view only exposed counts rather than individual names, the underlying misconfiguration (bypassing access controls entirely) is a serious architectural risk. If the view were ever expanded to include more detailed data, it could expose sensitive student activity information to peers.

What Was Fixed

The view was recreated with the correct security setting so that it now respects the platform's standard access controls. Administrators still see full viewing statistics as before. Regular students are only shown data relevant to their own activity.

4. Dangerous Files Could Be Uploaded Disguised as Images

High ✓ Fixed

What Was the Problem

Administrators can upload images for OSCE (clinical examination) questions. The platform was only checking the file's name extension (e.g., ".png") to decide whether to allow the upload — it was not checking what the file actually contained. A file's extension can be freely renamed by anyone.

Real-World Impact — What Could Have Happened

A malicious administrator, or an admin account that had been compromised, could rename a harmful file (such as a virus or malware) to end in ".png" and upload it to the platform. That file would then be stored on the platform's servers and potentially served to students. If a student downloaded or opened it, their device could be infected. Additionally, certain valid image formats (like SVG) can contain embedded scripts — these were also accepted, which could enable a different type of attack where simply viewing an "image" triggers malicious code.

What Was Fixed

The upload process now validates the actual file type (as reported by the browser's content detection) against a strict allowlist of safe image formats: JPEG, PNG, GIF, and WebP. Renaming a harmful file to ".png" will no longer work — the platform checks what the file really is, not just what it's called. SVG files are explicitly excluded due to their ability to carry executable code.
